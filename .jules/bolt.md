## 2024-03-07 - HTTPX AsyncClient Connection Pooling Optimization
**Learning:** Instantiating `httpx.AsyncClient()` inside a `with` block for every single request creates a significant performance bottleneck by bypassing connection pooling. It forces a new TCP connection (and potentially TLS handshake) on every API call. Furthermore, initializing `httpx.AsyncClient()` at the module/class level (at import time) instead of managing it within the application's lifecycle (e.g., inside FastAPI's `lifespan` event loop) is an anti-pattern that leads to `ResourceWarning: unclosed socket` warnings when the application shuts down or runs into event-loop binding issues.
**Action:** When making frequent HTTP requests via `httpx`, lazily initialize a shared `httpx.AsyncClient` inside a class method or context manager that executes within the running event loop. Ensure that the shared client is explicitly closed during the application's shutdown phase (e.g., using FastAPI's `lifespan` yield block) to prevent resource leaks.
